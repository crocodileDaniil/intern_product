# Commercial Offers Processing API
!!! Тут старая информация из ТЗ (пока читать не стоит)

Небольшой API-сервис на Python 3 для асинхронной обработки архивов с коммерческими предложениями.

## Как запустить

```bash
docker compose up --build
```

После запуска API будет доступен на `http://localhost:8000`.

OpenAPI-документация:

- Swagger UI: `http://localhost:8000/docs`
- OpenAPI JSON: `http://localhost:8000/openapi.json`

## Технологии

- **FastAPI** - быстрый способ сделать REST API и автоматически получить OpenAPI-документацию.
- **SQLAlchemy** - работа с PostgreSQL через ORM.
- **PostgreSQL** - хранение задач обработки и результата извлечения данных.
- **Docker Compose** - запуск приложения и базы данных одной командой.
- **unittest** - unit-тесты бизнес-логики без дополнительных зависимостей.

## Архитектура

Проект разделен на несколько простых слоев:

- `app/main.py` - HTTP endpoints.
- `app/auth.py` - простая ролевая авторизация по Bearer-токенам.
- `app/models.py` - ORM-модель задачи обработки.
- `app/services/parser.py` - бизнес-логика извлечения данных из архива.
- `app/services/tasks.py` - фоновая обработка задачи и обновление статуса.

Сценарий работы:

1. Пользователь с ролью `admin` или `specialist` загружает `.zip` архив в `POST /cp/tasks`.
2. Сервис создает запись в БД со статусом `pending`.
3. В фоне архив обрабатывается, статус меняется на `processing`.
4. После обработки статус становится `completed` или `failed`.
5. Любой авторизованный пользователь может получить задачу через `GET /cp/tasks/{task_id}`.

## Ролевая модель

Для тестового задания используется простая токенная модель:

- `admin-token` - роль `admin`, может создавать и читать задачи.
- `specialist-token` - роль `specialist`, может создавать и читать задачи.
- `user-token` - роль `user`, может только читать задачи.

Токен передается в заголовке:

```http
Authorization: Bearer specialist-token
```

## Формат архива

Сервис принимает только `.zip`. Внутри архива поддерживаются файлы:

- `.csv`
- `.json`
- `.txt`

Обязательные поля позиции:

- `supplier` - поставщик;
- `name` - название товара;
- `price` - цена.

Для CSV также поддерживаются альтернативные названия колонок:

- `supplier`, `vendor`, `company`
- `name`, `product`, `item`
- `price`, `cost`

TXT-формат строки:

```text
supplier;name;price
```

## Как проверить работу сервиса

Создать тестовый архив:

```bash
mkdir -p sample
printf 'supplier,name,price\nООО Ромашка,Ноутбук,1000.50\nООО Вектор,Ноутбук аналог,900\n' > sample/offers.csv
zip -j offers.zip sample/offers.csv
```

Создать задачу:

```bash
curl -X POST http://localhost:8000/cp/tasks \
  -H "Authorization: Bearer specialist-token" \
  -F "file=@offers.zip"
```

Получить задачу:

```bash
curl http://localhost:8000/cp/tasks/1 \
  -H "Authorization: Bearer user-token"
```

Ответ после обработки:

```json
{
  "id": 1,
  "file_name": "offers.zip",
  "status": "completed",
  "extracted_data": {
    "positions": [
      {
        "supplier": "ООО Ромашка",
        "name": "Ноутбук",
        "price": 1000.5,
        "source_file": "offers.csv"
      }
    ],
    "suppliers": ["ООО Ромашка"],
    "total_positions": 1,
    "min_price": 1000.5
  },
  "error": null,
  "created_at": "2026-04-16T10:00:00Z",
  "updated_at": "2026-04-16T10:00:01Z"
}
```

## Тесты

Unit-тесты проверяют бизнес-логику извлечения данных из архива:

```bash
python -m unittest
```

## Компромиссы

- Вместо настоящего ML/ИИ используется парсер-заглушка CSV/JSON/TXT, чтобы проверить работоспособность (как я понял ИИ уже это проект).
- Авторизация сделана через статические Bearer-токены, т.к. не было задачи по аутентификации и последующей генерации токена. В реальном проекте лучше подключить полноценную выдачу JWT и хранение пользователей.
- Таблицы создаются при старте приложения через SQLAlchemy, дальше актуальнее использовать Alembic
- Фоновая обработка сделана через `BackgroundTasks (нашёл самый простой). Для больших приложений лучше использовать celery и redis
- Вынести все необходимые конфигурации в .env файл и ключи , пока просто хард код.
